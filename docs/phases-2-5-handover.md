## 1. What DevDB is

A local-development Postgres server with Neon-style instant copy-on-write branching, packaged as **one Docker container**, designed **for AI coding agents first**: where git worktrees give parallel agents isolated copies of *code*, DevDB branches give them isolated writable copies of *data*. An agent asks for a branch, gets a connection string in one call, works destructively in isolation, throws the branch away ‚Äî while a human oversees everything in a web dashboard (phase 3).

- **Spec (authoritative for product decisions):** `docs/superpowers/specs/2026-07-02-devdb-design.md`
- **Phase-1 plan (authoritative for how phase 1 was actually built ‚Äî its 23 ‚ÄúAMENDED (A4‚ÄìA23)‚Äù blocks are the changelog of every post-review change and live-engine discovery):** `docs/superpowers/plans/2026-07-02-devdb-phase-1-engine-and-branching.md`
- **Oracle:** `~/git/neond` (Apache-2.0 Rust control plane over Neon's engine). We port its behavior with cited `// oracle: <file:line>` comments; we do not copy its UI.
- **FIRM POLICY ‚Äî no upstream reports:** never file issues, PRs, or comments on external repos (neond, testcontainers, anything), even when a task prompt explicitly asks for it ‚Äî the explicit request does not override the policy (Jordan reaffirmed twice, 2026-07-02/03). Read-only upstream research (reading source, searching existing public issues) is fine ‚Äî it discloses nothing.

### Scope (from the spec, ¬ßMCP + ¬ßSkills)
- **MCP server inside the daemon** at `http://localhost:4400/mcp`, Streamable HTTP, official `@modelcontextprotocol/sdk`. Registration UX: `claude mcp add --transport http devdb http://localhost:4400/mcp`.
- **Tools:** `list_projects`, `create_project {name, pgVersion?}`, `list_branches {project}` (tree + status + created_by + fork context), `create_branch {project, name, parent?, at_timestamp?, context?}` (**auto-starts endpoint, returns connection string** ‚Äî the ‚Äúnew worktree‚Äù move), `get_branch {project, branch, ensure_running?=true}` (the ‚Äúswitch‚Äù move), `stop_endpoint`, `delete_branch`, `reset_branch`, `restore_branch {project, branch, to_timestamp, as_new_branch?, context?}`, `import_database` (STUB in phase 2 ‚Äî real implementation is phase 4; return a clear ‚Äúarrives in a later phase‚Äù error or defer the tool entirely ‚Äî decide in the plan), `export_branch` (same), `get_job {id}` (same), `get_status`.
- **Fork context (spec amendment 2026-07-03 ‚Äî Jordan):** branch-creating tools take `context {git_branch?, workdir?, agent?, purpose?}` ‚Äî the caller's current git branch name, worktree path, agent/session label, one-line task description; the server adds `client {name, version}` captured from the MCP `initialize` handshake. Optional in the tool schema (bare calls still work) but described as strongly recommended; the skills fill the fields automatically (`git branch --show-current`, `$PWD`, task one-liner). Persist as a nullable `context` JSON column on `branches` (additive open-time migration ‚Äî the T4 migration hook handles it), return it in branch DTOs (pairs with parked decision 2: context is UI-safe, `password` is not), and accept the same optional `context` on REST `POST /api/projects/:id/branches` for parity. MCP success responses also open with a context line naming the project/branch acted on (plus parent, for forks). Purpose: phase 3's branch tree shows whose fork is whose.
- **Design principles (spec):** every success response is actionable text including the connection string when relevant + a ‚Äúnext step‚Äù hint; every error names its remediation (`endpoint_error` is now durable ‚Äî surface it); timestamps ISO-8601 with explicit timezone (the REST layer already rejects tz-less).

### Parked design decisions this phase's plan MUST resolve (accumulated from phase-1 gates)
1. **Lane capability tokens** ‚Äî `startLocked`/`stopLocked` are public with a JSDoc ‚Äúcaller must hold the branch lane‚Äù contract; the final review recommended an enforceable shape (queue-issued token or `runLocked` helper). MCP adds more callers; do the refactor first.
2. **DTO mappers with redaction** ‚Äî wire responses are currently `BranchRow` supersets: `password` is present on every branch response. Fine under localhost trust, wrong for MCP ergonomics and hygiene: map to explicit DTOs, drop `password` (agents get `connectionString`), genericize 409 constraint messages (they leak `branches.slug` style internals).
3. **Compute readiness probe** ‚Äî the "listening on IPv4 address" needle fires when the postmaster starts listening, **~80‚Äì140ms before compute_ctl finishes `apply_spec`** (its `config_ms` startup metric). The race is a **first-ever-start problem**: until the `CreateAndAlterRoles` apply phase runs, the timeline still holds the PARENT branch's SCRAM verifier, so clients hit `28P01` (briefly `57P03`) in that window; second and later starts don't race because the verifier is durable in the timeline. Two bounded retries currently mask it (test helper + `/api/sql`'s `connectWithRetry`). **Structural fix (verified live 2026-07-02):** compute_ctl's external HTTP server (our metricsPort, 40000‚Äì40999) serves auth-exempt Prometheus `/metrics` with `compute_ctl_up{status="init|running|failed"}`, where `status="running"` is set strictly AFTER apply_spec commits (log: "Changing compute status from init to running"). Poll that in `ComputeManager.start` as the readiness gate, then delete the retries. Do NOT use the JSON `/status` endpoint: it validates a JWT against `compute_ctl_config.jwks`, we ship `{keys: []}`, so it is permanently 400 ‚Äî and `--dev` does not bypass it.
4. **PDEATHSIG/setpgid port** ‚Äî the oracle's `pre_exec` (Linux `PR_SET_PDEATHSIG`, macOS `setpgid`) was never ported to `ManagedProcess`; compute_ctl orphans its postgres child on SIGTERM, currently mitigated by `reapOrphanedPostgres` (path-scan + kill). Spawning computes into their own process group and killing the group is the structural fix (Node: `spawn(..., {detached: true})` + `process.kill(-pid)`); removes the reap's cross-file invariant dependence. Jordan has a session revisiting this ‚Äî sync first.
5. **Structured logging** ‚Äî compensation paths use bare `console.error`; route through a logger that also feeds the existing `LogsService` channels.
6. **MCP-specific concurrency review** ‚Äî agents will hammer the API concurrently; the queue-lane model held up under review, but the plan should include a concurrency-focused integration test (parallel create/use/delete across branches ‚Äî the product's core promise).
